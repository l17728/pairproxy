### 开场

大家好，很高兴今天能来跟各位聊一聊。

我叫 XX，过去六年一直在做后端和基础设施。大约两个月前，我开始了一个实验：**能不能完全靠 AI，从零写出一个企业级的系统——而且全程不打开编辑器。**

44 天之后，我得到了一个 12.1 万行 Go 的项目，44 个版本，2894 个测试，全部通过。

今天我不打算聊技术选型，也不聊框架对比。我想聊一个更根本的问题：

**你把代码外包给 AI 了，那你做什么？你怎么对结果有信心？**

这是我在这 44 天里一直在想的事。我提炼出了五个控制点，接下来的二十分钟，我逐个拆开跟各位聊。

### 产品介绍

先说我们做了什么，再说怎么做的。

我们团队用 Claude Code，十几个人共享一个 Anthropic API Key。很快我就发现一个问题——谁用了多少 token，有没有超额，费用怎么分摊，完全不知道。

我就想，能不能做一个透明代理，架在 Claude Code 和 Anthropic 之间，把这些事全管起来。

这就是 PairProxy。架构很简单：一个 cproxy 跑在每个人本机，一个 sproxy 跑在服务器上。用户设两个环境变量，其他完全无感知。

这里有一个我觉得很重要的原则：**你练手的项目，一定要解决你自己的真实问题。** 因为只有你自己才知道什么叫"够好"。demo 谁都能糊弄过去，生产系统不行。

### 项目规模

来看几个数字。12.1 万行 Go、328 个文件、2894 个测试。

我想请大家注意一个数字：**测试文件 212 个，源码文件 116 个。测试文件比源码多了将近一倍。**

这不是我热爱写测试。而是在 AI Coding 的模式下，测试就是你的替代眼睛。你没法逐行 review 12 万行代码，但你可以让 2894 个测试替你看。

而且 0 失败——这不是巧合。后面我会讲为什么能做到。

大家可能觉得这个规模不算大。但我想说，重点不是规模，而是**这个规模下，我是怎么做到不用打开编辑器的**。

### 开发历程

这 44 天我分成了 7 个阶段。有一个很有意思的规律：**70% 的提交集中在前两周。**

这不是计划好的。这是我事后复盘发现的。

AI Coding 有一个天然节奏——前两周是"蜜月期"，代码库还很干净，AI 产出极快。但到了第三周、第四周，代码库开始有历史包袱了，bug 开始多了，测试开始要补了。这些事情 AI 做起来就没有前面那么利索。

所以我后来总结了一条经验：**如果你想用 AI Coding 做一个项目，把功能需求尽量集中在前两周提完。** 后面的时间留给稳定化和打磨。

别在第三周才开始想"我是不是该加个新功能"——那时候 AI 的产出效率已经开始下降了。

### CI/CD 流水线

这个流程图是我用 Mermaid 画的 CI/CD 流水线。git push 自动触发 build、vet、test、lint，打 tag 自动交叉编译 5 个平台，同时构建 Docker 镜像推到 ghcr.io。

我想特别说一下，**CI/CD 在 AI Coding 里不是"加分项"，是必需品。**

为什么？传统开发的时候，你写了代码，你手动跑一下测试，大致心里有数。但 AI Coding 里，你没看过代码。你的"心里有数"从哪来？

从 CI 来。CI 是你的第二双眼睛。每次 AI 提交，CI 替你跑 vet、跑 race detect、跑 lint。绿色通过了，你才能松一口气。

所以如果你今天就要开始用 AI Coding，第一件事不是写代码，是先把 CI 搭好。

### 转折

好，到这里我停一下。

刚才各位看到的——12 万行代码、44 个版本、CI/CD 流水线、Docker 镜像、Dashboard、多节点集群——

我想说一件事：**这 44 天里，我没有打开过编辑器。一次也没有。**

说这句话不是为了炫耀。我想表达的是：当你在 AI 周围搭了一套足够好的"脚手架"之后，它的产出是可以预测的、可以验证的、可以审计的。

问题是，这个脚手架长什么样？

接下来我把我提炼出的五个控制点拆开跟各位聊。

### 五个控制点概览

我把和 AI 协作的方式总结成了五个控制点。它们不是孤立的，是一条流水线：

你先告诉 AI 规则，再定义好任务，再控制执行过程，再验证结果，最后结构化验收。

这里我想说一个核心思想：**不是"信任 AI"，也不是"不信任 AI"，而是设计一个系统，让 AI 的对错变得可观测。**

什么意思？你不需要看每一行代码。但你需要知道：测试有没有过，CI 是不是绿的，验收报告里有没有暴露问题。

就好比你管理一个远程团队——你不盯着他们敲键盘，但你看产出、看指标、看报告。

---

### AGENTS.md

五个控制点里，**投入产出比最高的就是这一个——上下文管理。**

我写了一个文件叫 AGENTS.md，它不是项目文档，更像是给 AI 的"员工手册"。项目文档是给人看的，讲背景、讲道理。AGENTS.md 是给 AI 看的，约束行为、列规则。

举个例子。"请认真编写测试"——这句话你写进文档里，AI 根本不知道你要什么。但如果你写"配额系统出错时不阻断请求，只记日志，这叫 Fail-Open 原则"——这个 AI 能执行。

写完 AGENTS.md 之后，我每次 review 的改动量从大概 50% 降到了 15%。有一个反复出现的并发 bug，我把根因写进去之后，再也没复发过。

**各位回去如果只做一件事，我建议花两个小时写你的 AGENTS.md。** 不用完美，先写出来。

### Spec → Plan → TDD

这三个我合并讲，因为它们本质上是一条流水线：先想清楚要做什么，再想清楚怎么做，最后严格按步骤来。

Spec 里面有一个要素特别容易被忽略——"已拒绝方案"。AI 特别喜欢走捷径。如果你不告诉它"不要用方案 X，因为 Y"，它大概率会选 X。

然后是 TDD。TDD 的关键不是"先写测试"——关键是**先跑一遍测试，确认它确实失败了**。

为什么？因为 AI 写的测试，有可能永远不会失败。比如它把不该 mock 的东西 mock 了。如果你不验证失败，这个测试就是假的安全感。

我的原则很简单：**永远不要让 AI 直接开始写代码。** Spec → Plan → TDD，这个顺序不能跳。

### 六类失败模式

AI 会犯错，但有意思的是，它的错误是有规律的。我总结了六类，这里挑四个真实的讲。

第一个：**降级策略过严**。有一天 Claude Code 突然罢工了，我查了半天，发现 AI 把一个 thinking 参数原样转发给了老版本 API，老版本返回 400，AI 就认为这是不可恢复错误，直接中断了。但它其实只需要把那个参数去掉继续转发就行了。

第二个：**缓存安全假设**。JWT 做了缓存，命中之后直接放行。但有个用户已经被禁用了，缓存里还是"有效"，结果禁用操作 24 小时不生效。

我的经验是：**不要全量 review，你 review 不过来。** 对着这六类定向查就行——降级策略、缓存假设、数据库语义、算法副作用、边界条件、遗漏路径。

### 验收机制

说到验收，我想问各位一个问题：如果你的 AI 跟你说"所有测试通过"，你信吗？

"所有测试通过"——这句话你没法审计。你不知道它跑了几个测试，也不知道测试是不是真的在测该测的东西。

但如果它说 **"2,894 PASS, 0 FAIL"**——这句话你可以审计。

验收报告里我最看重的部分，不是"通过了什么"，而是 **"过程中发现并修复了什么"**。

如果一个 AI 做了自我检查，然后告诉你"一切顺利，没发现任何问题"——你反而应该警惕。要么是检查不到位，要么是隐瞒了问题。

"我发现 3 个问题并修复" 比 "一切顺利" 可信度高十倍。

### Harness 脚手架

我一直在说"脚手架"，这里展开讲讲到底是什么。

一个 Coding Agent，本质上等于 AI 模型加上你给它搭的 Harness。Harness 定义了 AI 能看到什么、能用什么工具、怎么被验证。

六个面：Context Files 注入知识、Tools/MCP 扩展能力、Sub-agents 做上下文隔离、Skills 按需加载、Hooks 做自动化流程、Back-pressure 做强制验证。

我想说一个可能反直觉的观点：**大多数人把时间花在"怎么跟 AI 说话"上——调 prompt、找技巧。但真正决定 AI 产出质量的，是你给它搭的 Harness。**

同样的模型，好的 Harness 和差的 Harness，产出质量天差地别。

---

### Skill 和记忆闭环

两个机制，一个管"知道什么"，一个管"记住什么"。

Skills 的核心思路是**渐进式知识披露**。别把所有规则一股脑塞进 AGENTS.md——context 有上限的。按需加载就好：碰到 PDF 任务加载 pdf skill，碰到邮件任务加载 email skill。

记忆闭环是我觉得整个方法论里**最有复利效应**的部分。流程是：踩坑 → 复盘归因 → 举一反三普查同类 → 写进 AGENTS.md 规则化 → 同类 bug 永不复发。

每个 bug 都是一次加强 Harness 的机会。**浪费 bug 就是浪费最贵的学习。**

我举一反三的方式不是随便看看——是搜相同模式，逐一确认，每个都补测试。这样下次 AI 在同一个地方就不会再犯了。

### AI 协作证据

这些都是 Git 历史里留下的真实痕迹。

有一个我觉得特别有意思：AI 对 AI 的批评。我让 AI 写了一份 335 行的 Review 文档，给初稿打了 55 分，然后又让另一个 AI 对这份 Review 本身打分——35 分。

这说明什么？**多层 AI 审核是真的有价值的。** 不是因为 AI 比人强，而是因为不同的 AI 在不同的上下文窗口里，看到的东西不一样。

还有一次，2 小时内改了 8 个模块——db、alert、api、proxy、cli、config——因为我让多个 agent 并行跑。

所以善用 Git 记录。你的 commit message、review 文档、验收报告，不只是过程证据，更是你复盘的原材料。

### 版本功能树

快速过一下版本演化。从 v1.0 的基础代理，到 v2.24 的企业级平台。

注意这个路径：v1.x 先把核心跑通——HTTP 代理、SSE 流式、Token 统计。v2.0 开始加企业特性——认证、配额、Dashboard。v2.9 做协议转换，这是最复杂的部分。v2.11 之后是集群、PostgreSQL、安全性。

为什么这么排？因为 **AI Coding 的项目也要增量演进**。不要让 AI 一次做太多。每个版本聚焦一个主题，做完验收再开下一个。这样出错的时候你能快速定位——"这个 bug 是 v2.9 引入的，因为那个版本在做协议转换。" 如果所有功能都是 v2.0 一起堆上去的，你根本无从查起。

### 三个反直觉洞察

接下来分享三个我在这 44 天里体会最深的洞察。可能跟你的直觉不太一样。

**第一个：减少 AI 的决策空间，比扩大 AI 的能力更有效。** 很多人觉得应该给 AI 更多自由度让它发挥。我的经验正好相反。当你用 Plan 把 AI 从"设计加实现"变成"按图执行加验证"，可预测性会大幅提升，返工率接近零。

**第二个：给 AI 的文档和给人的文档必须分开写。** 给人看的文档讲背景、讲道理。给 AI 看的文档约束行为、列规则。混在一起两头都失效。

**第三个：AI 的失误有规律可循。** 全量 review 成本太高，定向 review 六类就够。

### 总结

我们快速回顾三个数字。

44 天，121K 行——这是 AI Coding 的爆发力。

50% 降到 15%——这是我写完 AGENTS.md 之后，每次 review 需要改动的比例。

7 小时降到 0——这是一个并发 bug 写进 AGENTS.md 之后，再也没有复发过的时间。

各位回去如果只做一件事，我的建议是：**花两个小时写你的 AGENTS.md。**

问自己三个问题就行：有哪些 AI 不读 100 个文件就不知道的隐性规则？你踩过的最痛的三个坑，AI 知道吗？你的系统 Fail-Open 和 Fail-Closed 的边界在哪？

不用完美。先写出来。**迭代比完美重要得多。**

### 依赖陷阱

最后讲一个我心里一直在想的问题。

AI Coding 让你更快地构建了一个**你自己无法完全理解的系统**。

这个依赖是有曲线的。起步的时候感觉速度 5x，信心爆棚。然后你发现代码量已经大到逐行 review 不可能了。再然后你变成了 AI 的质检员。再然后你发现离开 AI 你已经无法维护了。最后 AI 的效率也从 5x 跌到 1.5x——因为代码库复杂到 AI 也开始犯错了。

这不是理论。我相信在座的很多人已经在第四阶段了。

好的 Harness 不能消除依赖。但它能把依赖控制在可审计的范围内：你不看代码，但你看测试结果。你不看每一行 diff，但你做定向 review。你不记得每个函数，但 AGENTS.md 替你记住了。

### 结束

最后，我想回到一句话：

> AI 是一个执行能力极强、但记忆极短、判断需要被约束的工程师。

你的工作不是写代码了。你的工作是**设计它的工作环境，同时保持自己理解这个环境的能力。**

一旦你不再理解 AI 在做什么——你就从管理者退化成了旁观者。

工具变了，但工程的基本规律没变。只不过我们对"工程"的定义变了：**不只是写代码的工程，更是管理 AI 写代码的工程。**

### Session 分析

最后一页，我想用数据来验证一下——刚才说的这些实践，到底长什么样？

我们分析了 57 个 session、834 条用户消息。结果很有意思。

占比最高的操作是什么？不是"做这个"、"改那个"。是 **"做完了吗？确定吗？"** ——占了 32%。第二高的是"跑测试"——23%。第三是"组织 review"——15%。

换句话说，**在 AI Coding 的模式下，你的角色从"写代码的人"变成了"验证代码的人"。**

大家看下面这个对比。典型用户是一句话描述需求，信任输出，修完即走。而我们的方式是：粘贴完整 Spec，每步验证，举一反三补测试，最后还要 commit、push、release、更新文档。
