### 提取方法

从 JSONL 格式的 session 文件中提取 `type=user` 且 `role=user` 的消息，过滤掉：
- `<local-command-*` 系统命令
- `<command-name>*` 命令消息
- `<task-notification>*` 后台任务通知
- `This session is being continued from a previous conversation` 上下文续接
- 纯 `hello` / `hi` 测试消息
- 长度 < 5 字符的噪声

### Session 文件结构

每个 JSONL 文件中包含以下 type：
- `permission-mode`: 权限设置
- `file-history-snapshot`: 文件快照
- `user`: 用户消息
- `assistant`: AI 回复
- `system`: 系统消息
- `progress`: 进度更新
- `queue-operation`: 队列操作
- `last-prompt`: 最后提示

## 三、10 条方法论原则（含原文证据）

### ① 质询确认 — 用"是否"推进确认（270 次）

核心行为：不给 AI 含糊空间，用封闭式问题迫使 AI 给出明确回答。

代表消息：
- `是否所有本次特性设计的需求全部实现，测试用例通过`
- `是否已经发布`
- `CI是否结束`
- `是否满足2000+测试用例全回归`
- `是否需要更新README，以及备注，手册，运维手册，API手册，以及changlog和 release notes等文档`
- `是否需要日志和测试`
- `是否补成足够充分的测试用例`
- `是否是一处修改导致全部报错返回不同的值，或者本来就有相同的问题。请研究一下`
- `是否通过所有的测试`
- `是否已经出新的release版本`

### ② 测试验证 — 每一步的验证门（194 次）

核心行为：每次改动后必问测试结果，不是"建议跑测试"而是"确认全部通过"。高频使用"继续直到全部通过"的循环指令。

代表消息：
- `回归所有2000+测试用例`
- `请继续，直到全部通过`
- `全量回归`
- `继续剩余工作`
- `是否补成足够充分的测试用例`
- `是否通过所有的测试`
- `运行全量测试`
- `请跑一下全量测试`
- `回归测试`
- `请继续下一轮，直到全部通过`（ChatLab 中出现 8+ 次）

### ③ Review / 检查 — 设计中就 Review（128 次）

核心行为：不是事后 review，而是**设计中就 review**。流程是 设计 → review → 修改 → 开发。

代表消息：
- `请组织专家对该设计进行review发现并修正问题，完善测试用例`
- `请评价一下之前的设计和review质量如何`
- `请检查是否实现了这个功能`
- `检视交付物尽量回顾本项目的开发过程`
- `请检查最新的release版本，文档是否刷新了，release notes写了没有`
- `请检查我们的代码是否实现了这个功能`
- `看看是否有类似的问题`（举一反三 + review 结合）

### ④ 文档更新 — 文档即流程（112 次）

核心行为：每次版本变更必同步文档，文档不是额外任务而是流程的一部分。

代表消息：
- `更新相应的文档`
- `更新到CLAUDE.md`
- `commit & push & release & 写release notes，notes中要说明新增了哪些重要的特性以及文档`
- `是否需要更新README，以及备注，手册，运维手册，API手册，以及changlog和 release notes等文档`
- `请新建一个目录，将以上讨论完整的写入文档，并整理出配套的模板`
- `更新相应的文档`（同一条在多个 session 中重复出现）
- `请检查最新的release版本，文档是否刷新了`

### ⑤ Bug 修复 — 定向修复（96 次）

核心行为：发现 bug 后定向修复，通常与举一反三结合。

代表消息：
- `在实现时按照建议，增加必要的告警，以方便不管是配置错误还是其他情况能够指出问题和可能的原因`
- `报告里有错误信息即可。如果已满足，请模拟测试一下`
- `错误和返回用户友好的提示`
- `修复这个错误`

### ⑥ Commit & Release — 版本纪律（92 次）

核心行为：一条消息同时要求 commit + push + release + notes，说明这是标准流程。

代表消息：
- `commit & push & release & 写release notes`
- `commit & push`
- `git push`
- `请帮我打个 tag v2.14.0 的 release commit 并写 release notes`
- `打 v2.14.0 的 release commit 并写 release notes`
- `发布到 GitHub 并创建 release`

### ⑦ 举一反三 — 发现一个 bug 普查全部（37 次）

核心行为：修复 bug 后必问"同类问题还有哪些地方"，并要求补测试。这是与普通用户最大的差异点之一。

代表消息：
- `每一个bug都要举一反三，对每一个问题补充分的测试用例`
- `每一个bug修复后都要举一反三，想一下为什么会出这个问题，并且检查是否有类似问题`
- `举一反三，查一下是否还有其他地方存在相同的问题`
- `发现一个bug，需要修复。此外每次worker启动都会提供服务的webUI的服务，会导致故障。举一反三。`
- `是否举一反三了，有没有类似的错误，并补齐测试`
- `举一反三，每一处类似的问题都找出来`
- `是否需要举一反三，全面的测试和回归`

### ⑧ Spec / Plan — 粘贴完整计划让 AI 执行（36 次）

核心行为：自己或让 AI 写计划 → 自己审核修改 → 粘贴给 AI 执行。绝不直接说"加个功能"。消息中经常出现 `Implement the following plan:` 后跟完整的 Markdown 计划文档。

代表消息：
- `Implement the following plan: # PostgreSQL Peer Mode（对等节点模式）计划 ...`（完整粘贴 300+ 行计划）
- `Implement the following plan: # PostgreSQL 支持计划 ...`
- `现在请启动开发，请制定计划和任务，然后完成。注意日志打印要完备，测试用例要充分。`
- `请组织专家对该设计进行review发现并修正问题，完善测试用例`

### ⑨ 统计数据 — 数据说话（22 次）

核心行为：用数字量化结果，不满足于"完成了"。

代表消息：
- `cloc统计规模`
- `是否有请求次数的统计，例如一共多少次请求，请求的分布情况，按照范围，按照TOP10的请求人`
- `要统计实际发送给LLM端点的有效载荷。请求体中的prompt是不大，但实际请求的文本比较大`
- `实际统计的数据是否实际发给那个LLM的统计`
- `用户使用统计`

### ⑩ CI 检查 — 交付的最后一道门（21 次）

核心行为：release 后必跟踪 CI，不等别人反馈。

代表消息：
- `CI是否结束`
- `查看一下 CI 测试结果`
- `查看 CI 结果`
- `查看ci展廊`
- `检查 CI 结果`
- `看 CI 状态`

## 五、上下文续接模式

多个 session 中出现 `This session is being continued from a previous conversation that ran out of context` 消息，用户通过 summary 机制无缝续接工作。

统计出现次数：PairProxy 中约 15 次，ChatLab 中约 12 次。

用户行为特征：
- Context 溢出是常态，不视为异常
- 通过自动 summary 保持连续性
- 续接后通常第一条消息就是继续上一个任务（如"继续剩余工作"）
- 从不因 context 丢失而放弃或重做

## 六、用户沟通风格

### 语言
- 主要使用中文
- 技术术语使用英文（commit, push, release, CI, test, review, plan, spec）
- 混合使用中英文，不刻意统一

### 简洁度
- 大部分指令非常简洁（"commit & push"、"CI是否结束"、"继续剩余工作"）
- Spec/Plan 消息则非常详细（粘贴完整 300+ 行 Markdown 计划）
- 两极分化：执行指令极简，规划文档极详

### 交互密度
- 平均每个 session 约 14 条有效消息
- 高密度 session（如 025ccfd6）可达 225 条 user 消息
- 密集使用"是否"质询模式（32% 的消息包含质询）

## 八、方法论核心总结

从 834 条消息中提炼的核心工作流：

```
设计 Spec → 专家 Review → 粘贴 Plan → AI 执行 → 测试验证 → CI 检查 → Commit & Release → 文档同步
                  ↑                                                         |
                  └───────── Bug 修复 → 举一反三 → 补测试 ←────────────────┘
```

关键原则：
1. **不信任，要验证** — 343 次验证操作（测试+CI+review）
2. **不模糊，要质询** — 270 次封闭式确认
3. **不遗漏，要文档** — 112 次文档同步
4. **不放过，要普查** — 37 次举一反三
5. **不手松，要纪律** — 92 次标准 commit/release 流程
