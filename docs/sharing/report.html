📊 PairProxy 分析报告

📈 Token 用量趋势?【用途】展示每日输入/输出/总 Token 的消耗量变化。 【看什么】Token 用量的增长或下降趋势，识别高峰日。 【怎么用】对比输入与输出占比，判断是否存在异常高输出场景（如长对话、大量补全）。

📊 每日请求数趋势?【用途】展示每天发起的 API 请求总量随时间的变化。 【看什么】工作日 vs 周末差异、异常峰值、整体增长率。 【怎么用】若某天请求量骤升，结合错误率图判断是否为批量异常请求。

🤖 模型每日用量趋势（堆叠面积）?【用途】按模型拆分，展示每天各模型 Token 消耗量的堆叠占比变化。 【看什么】哪个模型用量最高、是否有模型迁移趋势、某模型用量突增的日期。 【怎么用】面积越大表示该模型使用越多；若某模型面积快速扩大，需评估对应成本压力。

📐 I/O 比率每日趋势?【用途】展示每日输出 Token / 输入 Token 的比率变化。 【看什么】比率越高说明模型生成内容越多（长回复）；比率突增可能表示出现无限生成或配置异常。 【怎么用】正常值通常在 0.1–0.5 之间，超过 2 需排查是否存在异常请求。

⏱️ 延迟分布直方图?【用途】展示所有请求延迟的频次分布（直方图）。 【看什么】延迟集中在哪个区间，是否存在长尾（极慢请求）。 【怎么用】柱子越高表示该延迟区间的请求越多；若右侧有孤立高柱，说明有大量超时请求需排查。

🕐 24h×7天 请求热力图?【用途】以小时（0–23）× 星期几（周一至周日）的矩阵展示请求密度。 【看什么】哪个时间段使用最密集，是否有明显的上下班高峰规律。 【怎么用】颜色越深表示请求越多；可用于规划维护窗口（选用量最低的时段）。

🔥 Token 吞吐热力图?【用途】以相同的时间矩阵展示 Token 消耗总量的热力分布。 【看什么】哪些时段产生了最多的 Token 消耗（即成本压力最大的时段）。 【怎么用】与请求热力图对比——若某时段请求少但 Token 高，说明该时段存在大型请求（如长文档处理）。

🏆 TOP10 用户 Token 用量?【用途】列出消耗 Token 最多的前 10 名用户（条形图）。 【看什么】用量最高的用户是否符合预期，是否存在超出配额的使用。 【怎么用】结合配额表判断头部用户是否需要调整限额；用量异常高的用户需核查请求内容。

🔢 TOP10 请求人（按请求次数）?【用途】列出发起请求次数最多的前 10 名用户。 【看什么】高频请求用户是否与高 Token 用户重合，若不重合说明存在大量短请求（如自动化脚本）。 【怎么用】高频低 Token 用户可能是自动化工具，需评估是否符合使用规范。

👥 用户分层分布?【用途】将用户按 Token 消耗量分为超级用户、活跃用户、普通用户、低频用户四层。 【看什么】各层级用户数量与 Token 占比，判断使用集中度。 【怎么用】若超级用户（<5%）贡献了 >80% Token，说明依赖度过高，需评估风险。

📊 Token 用量百分位数 (P50–P99)?【用途】展示所有用户 Token 消耗量在 P50/P75/P90/P95/P99 分位数处的值。 【看什么】用户间的用量差异程度，P99 与 P50 差距越大说明分布越不均匀。 【怎么用】P95 值可作为配额上限的参考基准；P99 异常高时需排查极端用户。

📦 用户请求数分布（箱线统计）?【用途】用箱线图统计所有用户请求次数的分布（最小值、Q1、中位数、Q3、最大值）。 【看什么】请求次数的中位数与极值，判断用户行为的集中程度。 【怎么用】上边界（Max）远大于 Q3 时，说明存在极高频用户；中位数偏低说明大部分用户使用较少。

🤖 模型使用分布?【用途】以饼图展示各模型在请求数量上的占比。 【看什么】哪个模型是主力，是否有模型占比过高带来单点风险。 【怎么用】若某模型占比 >80%，需评估其稳定性与成本；占比极低的模型可考虑是否继续保留。

🌐 上游端点状态?【用途】展示各上游端点的请求量、平均延迟和错误率（条形图）。 【看什么】哪个上游服务响应最慢、错误率最高。 【怎么用】错误率 >5% 的端点需优先排查；对比延迟判断是否需要切换或降权某上游。

🥧 上游请求占比?【用途】以饼图展示各上游端点承接的请求数量占比。 【看什么】流量是否均衡分布在各上游，是否存在流量倾斜。 【怎么用】若某上游占比异常高，检查路由策略；配合上游状态图判断高占比端点的健康状况。

📈 上游延迟趋势（多线）?【用途】展示各上游端点的平均延迟随时间的变化趋势（多条折线）。 【看什么】各上游延迟是否同步波动（系统性问题）还是单独上升（该上游特有问题）。 【怎么用】某上游延迟持续高于其他时，考虑降低该上游权重或切换；若所有上游同时升高，排查网络层或模型提供商的全局问题。

📊 状态码分布?【用途】统计各 HTTP 状态码的请求数量（2xx 成功、4xx 客户端错误、5xx 服务端错误）。 【看什么】错误状态码的占比，哪类错误最常见。 【怎么用】429（限流）多 → 考虑提升配额或限速策略；401/403 多 → 检查认证配置；5xx 多 → 排查上游服务稳定性。

🏢 分组对比?【用途】按用户分组（部门/团队）对比各组的请求数、Token 消耗和费用。 【看什么】各组之间的用量差异，哪个组是最大消费方。 【怎么用】结合配额使用表，判断哪些组需要调整限额；用量远超其他组的团队可能存在批量自动化请求。

⏱️ 延迟箱线图（按模型）?【用途】按模型分组，用箱线图展示延迟分布（Q1/中位数/Q3/须）。 【看什么】哪个模型延迟最稳定（箱体窄）、哪个模型有严重长尾（须长）。 【怎么用】箱体上边界（Q3）高 → 该模型稳定性差；须末端极长 → 存在偶发超时，需设置合理超时阈值。

📊 P50/P95/P99 延迟趋势?【用途】展示每日 P50（中位数）、P95、P99 延迟的变化趋势。 【看什么】P95/P99 与 P50 的差距，差距越大说明尾部延迟越严重；趋势上升需警惕。 【怎么用】SLA 通常以 P95 或 P99 为准；若 P99 持续超过目标值，需优化慢请求路径。

⏱️ 每日平均/最大延迟趋势?【用途】展示每日平均延迟与最大延迟的双线趋势图。 【看什么】平均延迟代表整体水平，最大延迟暴露单日最差情况。 【怎么用】若最大延迟某天骤增但平均值正常，说明存在单次极端慢请求；两者同步上升说明系统整体变慢。

👤 用户请求频次分布?【用途】将用户按请求频次分桶（如 1–10次、11–50次、51–200次...），展示每桶用户数。 【看什么】大多数用户属于哪个频次区间，是否存在大量极高频用户。 【怎么用】若高频桶（>500次/天）用户数增多，需检查是否为自动化脚本；低频用户占比过高说明产品使用频率偏低。

🔬 延迟 vs Token 散点图（采样 1000）?【用途】以散点图展示单次请求的 Token 总量与延迟的关系（随机采样 1000 条）。 【看什么】延迟是否与 Token 量正相关，是否存在大量"高 Token 低延迟"或"低 Token 高延迟"的异常点。 【怎么用】正常情况下两者正相关；若高延迟点聚集在低 Token 区，说明存在与请求体积无关的系统性延迟（如冷启动、网络问题）。

💧 Input vs Output Token 相关性?【用途】散点图展示每次请求的输入 Token 与输出 Token 数量的关系。 【看什么】输入与输出是否有稳定比例关系，是否存在异常的极大输出（输入很少但输出极多）。 【怎么用】点群偏向右下（高输入低输出）→ 内容总结类场景；偏左上（低输入高输出）→ 内容生成类场景；离群点需核查是否为异常请求。

📈 帕累托图（累积贡献%）?【用途】展示用户按 Token 消耗降序排列后的累积贡献百分比（二八法则验证）。 【看什么】前 N% 的用户贡献了多少 Token，是否符合 80/20 规律。 【怎么用】若前 20% 用户贡献 >90% Token，资源重心应放在服务好头部用户；同时需评估尾部用户的使用激励策略。

📊 DAU/WAU/MAU 趋势?【用途】展示每日/每周/每月活跃用户数（DAU/WAU/MAU）的变化趋势。 【看什么】用户活跃度是否在增长，节假日/周末的活跃度规律。 【怎么用】DAU/MAU 比值（粘性指数）越高说明用户回访越频繁；MAU 增长但 DAU 不增说明新用户多但留存差。

⏱️ 上游延迟箱线图（按模型）?【用途】按上游端点/模型分组，展示延迟四分位数分布（箱线图）。 【看什么】各上游之间的延迟稳定性差异，是否有某上游延迟异常偏高或波动剧烈。 【怎么用】箱体越窄说明该上游延迟越稳定；中位线高的上游建议降低路由权重或更换。

📊 用户采纳率?【用途】展示注册用户中有实际使用（发起过请求）的比例（采纳率）。 【看什么】有多少注册账号是真正活跃的，采纳率是否在提升。 【怎么用】采纳率低（<30%）说明大量账号闲置，需加强用户引导或培训；采纳率接近 100% 说明活跃度良好但可能需要开放更多名额。

🎯 DAU/MAU 粘性指数?【用途】以仪表盘形式展示 DAU/MAU 的比值（粘性指数），反映用户每月使用天数的密度。 【看什么】值越接近 1 表示用户每天都在使用（高粘性），越接近 0 表示偶尔使用。 【怎么用】行业平均约 0.15–0.25；>0.5 为高粘性产品；下降趋势需排查是否有功能体验问题。

📅 用户留存曲线?【用途】以队列分析方式展示新用户在后续各天/周的留存率（第 N 天还有多少人继续使用）。 【看什么】留存率下降的速度，Day 7/Day 30 留存率是关键指标。 【怎么用】留存曲线趋于平缓说明有忠实用户群；Day 1 留存极低（<30%）需优化新手引导体验。

🚨 错误请求明细?【用途】列出所有非 2xx 状态码的请求明细，支持按状态码和模型过滤。 【看什么】错误的时间分布、高频出错用户、错误集中的模型。 【怎么用】筛选 401 → 排查 API Key 失效；筛选 429 → 识别超限用户；筛选 5xx → 定位上游故障时间段。
