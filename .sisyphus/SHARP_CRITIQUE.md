# 犀利点评：初稿设计与 Review 的致命缺陷

> 这份文档不避讳地指出初稿和初稿 review 的问题，以及为什么改进版本采取了截然不同的方向。

---

## 初稿设计：一份看似完整但**根本不可行**的设计

### 核心问题：建立在谎言上的架构

初稿的整个架构都基于一个**错误的假设**：

> "SyncLLMTargets 会修改 sp.targets"

**现实**：代码中 `SyncLLMTargets` 从来不改 `sp.targets`。只有一个地方赋值 `sp.targets`：`newSProxy` 构造函数，仅此一次，永远不变。

这意味着：

1. **初稿声称的"并发安全问题"完全是虚构的**
   - 假设了一个不存在的并发写入
   - 基于这个假设推导了一系列"需要 atomic.Pointer"的结论
   - 实际上这些都是空中楼阁

2. **初稿设计的 F2、F3 根本无法工作**
   ```
   预期流程：
   WebUI 改 supported_models → DB 更新 → SyncLLMTargets → sp.targets 也更新 → 路由立即生效
   
   实际流程：
   WebUI 改 supported_models → DB 更新 → SyncLLMTargets → sp.targets 没变 → 路由仍用旧值 ❌
   ```
   
   **必须重启服务**才能生效。这是功能 fail，不是 BUG。

### F1 的目标被设计本身违背了

初稿 F1 说：
> "配置文件只播种不覆盖，WebUI 修改永久生效"

但 F2、F3 的设计中，一旦 WebUI 修改了新增的 `supported_models` 和 `auto_model` 字段：

```go
// 初稿的实现逻辑（伪代码）
func supportedModelsForURL(url string) []string {
    for _, t := range sp.targets {  // ← sp.targets 从不被 SyncLLMTargets 更新
        if t.URL == url {
            return t.SupportedModels  // ← 仍是旧值
        }
    }
}
```

**结果**：需要重启服务才能看到 WebUI 的修改。这直接**违反了 F1 的承诺**。

### 这是可以立即发现的缺陷

只需一个问题就能暴露这个缺陷：

> 在 serveProxy 中，supportedModelsForURL 查的是什么数据源？

**答案决定了一切**：
- 如果查 `sp.targets`（初稿这么做了）→ 不支持热更新 ❌
- 如果查 `sp.llmBalancer.Targets()`（改进版这么做）→ 支持热更新 ✅

初稿的设计者没问这个问题。

## 初稿 Review：一份**完全失败**的审查

初稿的 review 质量甚至比设计本身更差。它做了一件很有趣的事：

**发现和强调了一些完全不存在的问题，同时忽视了真实的致命缺陷。**

### 问题 1：虚构的并发竞态

初稿 review 用了大量篇幅讨论：

> "sp.targets 并发安全问题"

基于的"事实"：
> "SyncLLMTargets 中直接赋值 sp.targets = newTargets，没有 mutex 或 atomic 保护"

**这完全是编造的**。代码中根本不存在这行代码。

然后 review 在此基础上推导出：
- 需要用 `atomic.Pointer`
- 需要 O(1) lookup 优化
- 需要三个 race condition 测试（T7）

**所有这些都是浪费时间**。

这不仅是"误诊"，而是**做了一个彻底的诊断错误，然后围绕这个错误做了一大堆推导**。

### 问题 2：触发条件不可能的问题

初稿 review P3 说：

> "如果管理员通过 WebUI 编辑 config-sourced target 并修改 Source 为 database"

**WebUI 根本做不到这个**。代码中：

```go
// handleUpdateTarget 第一件事
if !target.IsEditable {
    http.Error(w, "...", http.StatusForbidden)
    return  // ← 直接返回，无法修改任何字段
}
```

WebUI 对 config-sourced targets 直接返回 403，无法修改任何字段，更不用说修改 `Source` 字段。

这是一个"基于不可能的前提条件的问题"——是杜撰出来的。

### 问题 3：误解的参数问题

初稿 review P4 说：

> "supportedModelsForURL(t.ID) 是 bug，因为 t.ID 不是 URL"

**代码中明确有**：
```go
lb.Target{
    ID: t.URL,  // ← ID 就是被赋值为 URL
    ...
}
```

所以 `t.ID` 的值就是 URL 字符串。不是 bug。

Review 甚至没看这一行代码就下结论了。

### 问题 4：遗漏真实的关键缺陷

与虚构这三个问题相反，review **完全遗漏了**：

1. **`sp.targets` 静态导致热更新不可行** ← 这是最严重的架构缺陷
2. **两级过滤在闭包内混合不可行** ← 这会导致 fail-open 逻辑错误
3. **buildRetryTransport 缺少 requestedModel** ← 这是隐藏的 bug

这三个都是真实的、会导致功能失败的问题。但 review 完全没提。

### Review 失败的根本原因

review 没有**验证设计的基本假设**。它做的是：

1. 读设计文档的描述
2. 基于描述做推理
3. 给出问题

但从不问："设计文档说的是真的吗？"

如果问这个问题：
- "SyncLLMTargets 真的会改 sp.targets 吗？" → 查代码 → 不会 ❌
- "WebUI 真的可以修改 Source 字段吗？" → 查代码 → 不行 ❌
- "t.ID 真的不是 URL 吗？" → 查代码 → 其实是 ❌

**每个虚假的问题都可以在 10 秒内通过查代码发现。**

但 review 没这么做。

---

## 更深层的问题：为什么会这样？

### 设计者的失败

**失败原因**：设计者没有充分理解代码。

设计者看到 `SyncLLMTargets` 这个函数名，想当然地认为它会做一些"同步"的事。但从不读实现，不知道它实际上只更新 `llmBalancer`。

这反映了：
1. **快速迭代的压力** - 急着写设计，没时间读代码
2. **缺乏可问性的架构** - 如果架构清晰，可以从名字推导；但如果有"静态字段"这样的隐含设计，就需要读代码才知道
3. **缺少整体验证** - 设计者从不问"WebUI 改动真的能立即生效吗？"

### Review 者的失败

**失败原因**：Review 者采取了"形式化"的 review 方法。

Review 的模式是：
1. 读设计文档的每个章节
2. 问"这里有没有问题"
3. 列出问题

但从不：
1. 问"设计的基本假设是对的吗"
2. 验证假设对应的代码
3. 对标功能目标检查可行性

这反映了：
1. **缺乏代码驱动的思维** - Review 是基于设计文档而非代码
2. **缺少目标导向** - 没有问"这个设计是否真的能达成 F1、F2、F3 的目标"
3. **过度信任设计文档** - 假设设计文档的描述是准确的

## 改进版为什么成功了？

改进版采取了**完全不同的方法论**：

### 1. 深度代码验证

第一步不是读设计，而是用 Agent 读取关键代码：

```
- SyncLLMTargets 实际做什么？
- sp.targets 被谁修改？
- lb.Target 包含哪些字段？
- buildRetryTransport 怎么调 pickLLMTarget？
```

**基于代码事实做推导，而非基于设计文档的推断**。

### 2. 对标功能目标

每个功能都问一个基本问题：

> F1 要求"WebUI 改动永久生效"，当前设计真的能做到吗？

追问这个问题，立即暴露了 `sp.targets` 静态这个致命缺陷。

### 3. 优先级的正确调整

在改进版中：
- P1、P2、P3 是**真实的、致命的**架构问题
- 初稿虚构的 race condition 被删除
- 初稿遗漏的真实问题被补回来

## 对数据驱动 Review 的启示

### 错误的 Review 方法

```
设计文档 → 推理 → 列问题 ❌
```

这个模式会导致：
- 虚构不存在的问题（因为推理可能错）
- 遗漏真实的问题（因为你在读的不是代码）

### 正确的 Review 方法

```
设计文档 → 代码验证 → 功能目标对标 → 列问题 ✅
```

这个模式确保：
1. 问题都是基于代码事实，不是推理
2. 问题都与功能目标相关，不是形式化的
3. 问题的优先级清晰

---

## 你会在其他地方看到这个问题

这种"设计-review 脱节"的问题在实际工程中很常见：

### 场景 1：API 设计 Review
- 设计者声称"API 支持热更新"
- Review 者没验证"配置改后是否真的能看到新值"
- 实现时才发现数据源是静态缓存

### 场景 2：并发 Review
- 设计者声称"需要原子操作保护"
- Review 者相信并深入讨论并发策略
- 实际上并发写入根本不存在

### 场景 3：性能 Review
- 设计文档描述了优化策略
- Review 者讨论性能指标
- 从不问"这个优化真的会应用到最常见的代码路径吗"

**关键教训**：在深入讨论细节前，先验证基础假设。

## 对初稿设计者的建议

1. **在动笔前读代码**
   - 特别是关键的逻辑（如 SyncLLMTargets、pickLLMTarget）
   - 理解"哪些数据是静态的，哪些是动态的"

2. **问一个关键问题**
   - "用户做了改动后，系统是否能立即看到效果？"
   - 这决定了很多架构决策

3. **端到端地验证**
   - 从用户操作（WebUI 改 supported_models）
   - 到系统响应（下一次路由时使用新值）
   - 中间的每一步都要确认

4. **写下关键架构决策**
   - "数据从哪里来"（sp.targets vs sp.llmBalancer.Targets）
   - "何时更新"（启动 vs 热更新）
   - 这些不是细节，是核心

---

## 对初稿 Review 者的建议

1. **不要只读设计文档**
   - 设计文档可能有错
   - 代码是真相

2. **验证关键假设**
   - 设计文档说"SyncLLMTargets 更新 sp.targets"？→ 查代码
   - 设计文档说"WebUI 可以修改 X 字段"？→ 查代码
   - 设计文档说"有并发竞态"？→ 查代码

3. **对标功能目标**
   - 不要讨论架构细节，先问"能达成目标吗"
   - 如果目标无法达成，再多细节优化都是浪费

4. **优先级排序**
   - 致命缺陷（功能无法实现）> 真实的 bug > 优化建议 > 形式问题
   - 不要因为虚构的问题占据篇幅而忽视致命缺陷

---

## 最后的话

这个案例展示了为什么**代码驱动的架构设计和 review 至关重要**。

初稿设计者和 review 者都不是坏人，只是采取了错误的方法：
- 设计者相信名字和推理
- Review 者相信设计文档

结果是：完成度 100%、看起来专业的一份文档，但**根本不可行**。

改进版通过回到代码和功能目标，识别出了真实的问题，提出了真正可行的方案。

**这就是为什么"与其写更多文档，不如先读代码"。**
